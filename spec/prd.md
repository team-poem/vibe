# [PRD] V.I.B.E

> Volume-Initiated Background Establisher

## 1. 개요

- **프로젝트명:** V.I.B.E
- **제품 형태:** macOS 데스크톱 앱
- **목적:** 사용자가 설정한 물리적 트리거(초기 버전은 이중 박수)를 감지해, 미리 등록한 여러 macOS 제어 동작을 빠르게 실행하는 자동화 앱을 만든다.
- **핵심 가치:** 키보드, 마우스, 단축키 조작 없이도 사용자가 원하는 작업 환경을 즉시 호출할 수 있게 한다.
- **핵심 방향:** 앱은 항상 빠르게 반응해야 하며, 사용자가 컴퓨터를 켤 때마다 별도 조작 없이 백그라운드에서 준비되어 있어야 한다.

## 2. 문제 정의

사용자는 매번 작업을 시작할 때 여러 앱을 켜고, 브라우저 탭을 열고, 음악을 재생하고, 볼륨이나 창 상태를 조정한다. 이 과정은 반복적이고 사소하지만 집중을 시작하기 전의 마찰을 만든다.

V.I.B.E는 이 반복 과정을 하나의 트리거로 실행한다. 초기 트리거는 이중 박수이며, 이후 버전에서는 다른 트리거 방식도 확장할 수 있는 구조를 목표로 한다.

## 3. 목표

- 사용자가 데스크톱 앱에서 자동화 루틴을 직접 설정할 수 있다.
- 사용자가 컴퓨터를 켜면 V.I.B.E가 자동으로 실행되고 백그라운드에서 대기한다.
- 이중 박수 감지 후 등록된 루틴이 체감상 즉시 실행된다.
- 앱 실행, URL 열기, 음악 제어, 볼륨 조정, 스크립트 실행 등 여러 제어 동작을 조합할 수 있다.
- 특정 예시에 종속되지 않고, 사용자가 원하는 작업 세팅을 자유롭게 구성할 수 있다.

## 4. 비목표

- 초기 버전에서 모든 운영체제를 지원하지 않는다. 우선 macOS만 대상으로 한다.
- 초기 버전에서 복잡한 음성 명령 인식은 구현하지 않는다.
- 초기 버전에서 완전한 노코드 자동화 빌더까지 제공하지 않는다. 단순하고 빠른 루틴 설정을 우선한다.
- 보안상 위험한 시스템 제어를 무제한 허용하지 않는다. 사용자가 명확히 설정한 액션만 실행한다.

## 5. 타겟 환경 및 제약 조건

- **운영체제:** macOS
- **아키텍처:** Apple Silicon 및 Intel Mac 지원
- **권한 요구:**
  - 마이크 접근 권한
  - 손쉬운 사용(Accessibility) 권한
  - Mac 시작 시 자동 실행 권한
  - 필요 시 자동화(Automation) 권한
- **입력 장치:** 내장 마이크 또는 외장 오디오 입력 장치
- **실행 형태:** 메뉴바 또는 트레이에 상주하는 데스크톱 앱

## 6. 사용자 시나리오

### 6.1. 최초 설정

1. 사용자가 V.I.B.E 앱을 실행한다.
2. 앱이 필요한 macOS 권한을 안내한다.
3. 사용자가 마이크 권한과 손쉬운 사용 권한을 허용한다.
4. 사용자가 "Mac 시작 시 자동 실행" 옵션을 켠다.
5. 사용자가 하나 이상의 자동화 루틴을 만든다.
6. 사용자가 이중 박수 테스트를 실행해 감도와 오탐지 수준을 확인한다.

### 6.2. 일반 사용

1. 사용자가 Mac을 켠다.
2. V.I.B.E가 자동 실행되어 백그라운드에서 대기한다.
3. 사용자가 박수를 두 번 친다.
4. 앱이 이중 박수 패턴을 감지한다.
5. 앱이 사용자가 설정한 루틴을 순서대로 실행한다.
6. 사용자는 별도 조작 없이 원하는 작업 환경에 진입한다.

## 7. 기능 요구사항

### 7.1. 데스크톱 앱

- 앱은 macOS 데스크톱 앱 형태로 제공한다.
- 앱은 메인 설정 화면을 제공한다.
- 앱은 백그라운드 대기 상태를 지원한다.
- 앱은 메뉴바 또는 트레이에서 현재 상태를 확인할 수 있어야 한다.
- 사용자는 메뉴바 또는 트레이에서 감지 기능을 켜고 끌 수 있어야 한다.
- 사용자는 앱에서 현재 마이크 입력 상태와 트리거 감지 상태를 확인할 수 있어야 한다.

### 7.2. Mac 시작 시 자동 실행

- 사용자는 앱 설정에서 "컴퓨터 시작 시 자동 실행" 옵션을 켜고 끌 수 있어야 한다.
- 옵션이 켜져 있으면 Mac을 켠 뒤 V.I.B.E가 자동으로 실행되어야 한다.
- 자동 실행된 앱은 사용자를 방해하지 않도록 기본적으로 백그라운드 또는 메뉴바 상태로 시작한다.
- 자동 실행 실패 시 사용자가 원인을 파악할 수 있도록 상태 메시지를 제공한다.

### 7.3. 오디오 트리거 엔진

- 앱은 백그라운드에서 마이크 입력 레벨을 실시간으로 감시한다.
- 초기 버전은 이중 박수(Double-Clap) 감지를 기본 트리거로 제공한다.
- 이중 박수는 다음 조건을 기반으로 판별한다.
  - 설정된 임계값(Threshold) 이상의 피크가 발생해야 한다.
  - 첫 번째 피크와 두 번째 피크 사이의 간격이 설정 범위 안에 있어야 한다.
  - 너무 짧거나 긴 간격은 무시한다.
  - 연속 소음 또는 긴 소음은 박수로 판단하지 않는다.
- 사용자는 감도, 최소 간격, 최대 간격을 설정할 수 있어야 한다.
- 앱은 테스트 모드를 제공해 사용자가 자신의 환경에 맞는 값을 찾을 수 있게 한다.
- 오탐지를 줄이기 위해 단발성 충격음, 기침, 문 닫힘 소리 등은 최대한 무시해야 한다.

### 7.4. 액션 및 루틴 시스템

`3.2`의 작업 세팅 자동화는 고정된 기능 목록이 아니라, 사용자가 구성 가능한 액션 시스템의 예시다. 앱은 특정 작업 환경에 종속되지 않고 여러 액션을 조합할 수 있어야 한다.

지원 액션 예시는 다음과 같다.

- **앱 실행:** Cursor, Chrome, Safari, Spotify 등 지정한 앱을 실행한다.
- **URL 열기:** 지정한 URL 목록을 기본 브라우저 또는 선택한 브라우저로 연다.
- **브라우저 탭 열기:** Google, TradingView, YouTube 등 여러 페이지를 한 번에 연다.
- **음악 제어:** 지정한 음악 서비스 또는 URL을 열고 재생을 유도한다.
- **시스템 볼륨 조정:** 루틴 실행 전후로 볼륨을 설정한다.
- **Shell Script 실행:** 사용자가 등록한 안전한 셸 명령을 실행한다.
- **AppleScript 실행:** macOS 앱 제어가 필요한 경우 AppleScript를 실행한다.
- **Shortcuts 실행:** `shortcuts run "VIBE_Routine"` 형태로 macOS 단축어를 실행한다.

루틴은 여러 액션의 묶음이다.

- 사용자는 루틴 이름을 설정할 수 있어야 한다.
- 사용자는 액션의 실행 순서를 설정할 수 있어야 한다.
- 사용자는 액션별 지연 시간 또는 병렬 실행 여부를 설정할 수 있어야 한다.
- 기본 루틴 예시는 다음과 같다.
  - Cursor 실행
  - Chrome에서 Google, TradingView, YouTube 열기
  - 시스템 볼륨 설정
  - 음악 재생 URL 열기

### 7.5. 설정 저장

- 사용자의 루틴, 트리거 감도, 자동 실행 여부는 로컬에 저장한다.
- 설정은 앱 재시작 후에도 유지되어야 한다.
- 설정 파일이 손상된 경우 기본값으로 복구하고 사용자에게 알린다.
- 초기 버전에서는 클라우드 동기화를 제공하지 않는다.

### 7.6. 상태 및 피드백

- 앱은 현재 상태를 명확히 표시해야 한다.
  - 대기 중
  - 마이크 권한 없음
  - 트리거 감지됨
  - 루틴 실행 중
  - 루틴 실행 완료
  - 루틴 실행 실패
- 루틴 실행 결과는 간단한 로그로 확인할 수 있어야 한다.
- 실패한 액션이 있으면 어떤 액션에서 실패했는지 표시해야 한다.

## 8. 성능 요구사항

이 프로젝트는 반응 속도가 매우 중요하다. 사용자가 박수를 친 뒤 루틴이 늦게 실행되면 제품 가치가 크게 떨어진다.

- 앱은 백그라운드 대기 중에도 가볍게 동작해야 한다.
- 마이크 입력 분석은 낮은 지연 시간으로 처리해야 한다.
- 이중 박수 감지 후 첫 액션 실행까지의 목표 시간은 300ms 이내로 한다.
- 전체 기본 루틴 실행 시작은 1초 이내에 체감되어야 한다.
- 백그라운드 상태에서 CPU 사용량은 가능한 한 낮게 유지한다.
- 오디오 분석 루프는 UI 렌더링과 분리해 UI 지연이 트리거 감지에 영향을 주지 않도록 한다.
- 앱 시작 시간은 짧아야 하며, Mac 시작 시 자동 실행되어도 사용자를 방해하지 않아야 한다.

## 9. 기술 방향

기술 스택은 구현 편의성과 성능을 기준으로 선택한다. 단, 데스크톱 앱 형태와 빠른 반응성을 최우선으로 둔다.

### 9.1. 후보 기술

- **Desktop Framework:** Electron 또는 Tauri
- **Frontend:** React, TypeScript
- **Native/Backend Layer:** Node.js, Rust, Python 중 성능과 macOS 연동에 적합한 방식 선택
- **Audio Processing:** 실시간 오디오 입력을 안정적으로 처리할 수 있는 네이티브 라이브러리 또는 런타임 사용
- **macOS Control:** AppleScript, Shell Script, Shortcuts, macOS Login Items API
- **Local Storage:** JSON 파일, SQLite, 또는 앱 설정 저장소

### 9.2. 구현 원칙

- UI와 트리거 감지 엔진을 분리한다.
- 트리거 감지 엔진은 가능한 한 가볍고 빠르게 유지한다.
- 액션 실행부는 확장 가능한 구조로 설계한다.
- 루틴 실행은 불필요하게 순차 처리하지 않고, 병렬 실행 가능한 액션은 병렬 실행한다.
- 권한 요청과 시스템 제어는 사용자에게 명확히 설명한다.
- 초기 구현에서는 복잡한 추상화보다 빠르게 검증 가능한 구조를 우선한다.

## 10. 보안 및 권한

- 앱은 마이크 입력을 외부 서버로 전송하지 않는다.
- 모든 오디오 분석은 로컬에서 처리한다.
- 사용자가 등록한 스크립트는 로컬에서만 실행한다.
- 위험한 액션 실행 전에는 사용자가 명확히 설정했는지 확인한다.
- 권한이 없을 때는 기능을 조용히 실패시키지 않고, 필요한 권한과 해결 방법을 안내한다.

## 11. MVP 범위

- macOS 데스크톱 앱 생성
- 메뉴바 또는 트레이 상주
- Mac 시작 시 자동 실행 옵션
- 마이크 권한 요청 및 상태 표시
- 이중 박수 감지
- 감도 및 간격 설정
- 루틴 생성, 수정, 삭제
- 앱 실행 액션
- URL 열기 액션
- Shell Script 또는 Shortcuts 실행 액션
- 루틴 실행 로그

## 13. 성공 기준

- 사용자가 최초 설정 후 Mac을 다시 켜도 앱이 자동으로 대기 상태에 진입한다.
- 사용자가 이중 박수를 치면 등록된 루틴이 안정적으로 실행된다.
- 기본 루틴 기준으로 박수 감지 후 1초 이내에 앱 실행 또는 URL 열기가 시작된다.
- 사용자는 개발 루틴, 차트 모니터링 루틴, 음악 루틴 등 여러 목적의 루틴을 직접 구성할 수 있다.
- 앱은 백그라운드에서 동작해도 사용자가 느낄 정도의 성능 저하를 만들지 않는다.
